In reply to this message
In our implementation of dlcBTC, the "Bitcoin oracle" has been replaced with Ethereum blockchain and a DLC Attestor. The Attestor receives a signal from Eth, verifies it on-chain, and then publishes it's "attestation", which is a number that it generates using its private key.

For example, dlcBTC as a wrapped token does not require a price feed oracle. The user locks into the DLC, Eth mints dlcBTC. If there's an unmint, Eth sends the signal to the attestor.

In the future, we plan to launch a stablecoin (dlcUSD). In that case, we could use an oracle like Chainlink, that sends the signal to Eth, which sends to the attestor.
12:03
In reply to this message
Our original thought was that the counterparty to the DLC is a protocol like Uniswap. But protocols don't know how to integrate that, and they don't want to manage a BTC address or do DLC signing. So, DLC.Link will be the counterparty to the transaction, and the dlcBTC token will be listed on Uniswap
12:04
Right now, we run centralized DLC attestors. However, we are decentralizing it, by having node operators run them. So, the user does not have to trust us
12:05
In V1, the depositor (Alice) is locking with Alice. We are the counterparty, but in both DLC outcomes, BTC goes back to Alice. There is no scenario in which we can receive BTC funds
