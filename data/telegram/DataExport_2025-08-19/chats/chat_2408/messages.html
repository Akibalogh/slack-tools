Matthew ฿lack Atomic
Hey Jesse! Yea you’re right that description is very confusing (we should probably update the docs, PRs welcome 🙂 )

@node-dlc is actually completely written in TS, and has the logic for serialization / deserialization of all messages, as well as core finance logic for creating hyperbola payout curves (i.e. options). It does not contain wallet logic for actually constructing a DLC however.

@node-dlc is used within bitcoin-abstraction-layer (also known as BAL https://github.com/AtomicFinance/bitcoin-abstraction-layer) along side cfd-dlc which provides wallet functionality as well as logic for constructing a DLC.

You can see here in the integration tests where cfdDlcJs is imported from the wrapper (https://github.com/AtomicFinance/bitcoin-abstraction-layer/blob/master/tests/integration/common.ts#L23) and then passed into the BitcoinDlcProvider (https://github.com/AtomicFinance/bitcoin-abstraction-layer/blob/master/tests/integration/common.ts#L91)
14:11
Once those providers are setup, you can call functions to construct a dlc (i.e. createDlcOffer, etc.)
14:18
We’re also in the process of updating to the latest #163 serialization (all test vectors passing for DLC Offer messages, and we’re working on DLC Accept / Sign + updating finance modules as well as BAL https://github.com/AtomicFinance/node-dlc/tree/wip-dlc-specs-163)
