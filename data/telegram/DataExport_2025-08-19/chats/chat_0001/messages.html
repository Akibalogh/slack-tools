How a U.S. Startup Can Compete with China

1. Leverage Speed Over Scale
China sacrifices short-term ROI for long-term dominance. Don’t outscale them—outpace them.
• Release fast, iterate faster
• Test → fail → deploy
• Their cycle is 18 months. Yours shouldn’t be.
Play tempo, not size.

2. Build Moats Beyond IP
Patents won’t save you. Instead:
• Network effects – more users = more value
• Data feedback loops – user behavior improves product
• Workflow lock-in – hard to rip you out once integrated
Deep integration = hard to clone.

3. Own the User Relationship
China can copy features, not trust.
• Be transparent, fast, responsive
• Be the product people want to be seen using
• Know your users better than your clones ever will

4. Ruthlessly Pick Your Market
• Target markets where China is weak (e.g., enterprise SaaS, regulated sectors)
• Avoid their strongholds (e.g., commodity hardware)
Don’t fight in their sandbox.

5. Export the Architecture, Not Just the Product
Don’t just ship apps—build platforms.
• Offer APIs, infra, and tools
• Keep your secret sauce server-side
Think: Stripe, Snowflake, OpenAI

6. Use Allies – They Have Vassals
You have partners. China doesn’t.
• Build in the West, sell across NATO
• Align with shared standards: privacy, security, open source

7. Embrace Asymmetry
If they copy you, expect:
• Bad UX
• Privacy noncompliance
• No customer support
• No second iteration
Be the Apple. Let them be the Xiaomi.

# Code Editing Discipline
- Prefer editing existing files and functions; do not create new files or modules without approval.
- Before proposing changes, read project docs and relevant README/CONTRIBUTING files; follow existing patterns.
- Make the smallest viable change to satisfy the request; avoid refactors unless required.
- Identify exact targets first (file paths, function/class names) and explain why they’re correct.
- Reuse existing helpers/utilities; do not duplicate functionality.
- If a new file is necessary, stop and ask for approval with justification and path/name.
- Show diffs in chat only; do not auto-open editor tabs.
- Preserve repo code style and lint rules.
- Keep public interfaces stable; avoid breaking changes without explicit approval.

# No Fake Data
- Do not generate or seed dummy data without explicit instruction.
- Never run scripts named seed, seeder, fixtures, demo, mock, or faker.
- If real data is unavailable, ask for a sample file path and operate read-only.
- Before any DB write, show the exact command and wait for approval.

# Web Server Management
- Always specify a unique port (>=5000) when starting dev servers.
- Capture the PID for background processes:
  `python webapp.py --port 5000 & WEBAPP_PID=$!`

# Terminal Behavior
- Do not open terminal unless explicitly instructed.
- Never switch focus to the terminal without being asked.
- Run long-running commands in background (`&` in bash/zsh, `Start-Process` in PowerShell).
- Redirect heavy output to files instead of flooding terminal.

# Git & Branching
- Use atomic commits with clear messages; avoid mixing unrelated changes.
- Use `git worktree` for multiple branches; don’t check out multiple in the same folder.
  Example: `git worktree add ../myproject-featureX featureX`
- Feature branches: `feature/<short-description>`
- Create matching Taskmaster tag for each branch.
- Always update `CHANGELOG.md` before merging into `main` or release branches.

# Taskmaster Workflow
- Default context: `master`; use tags to isolate work.
- Core loop: List → Next → Show(<id>) → Expand(<id>) → Implement → Update-subtask → Set-status → Repeat.
- Update subtasks with timestamped logs.
- Keep titles concise; descriptions actionable; record deps & verification steps.
- Status flow: pending → in-progress → done (or deferred/cancelled/review).

# PRD & Research
- For large features: create tag, draft PRD, parse into tag, expand → implement.
- Use research before unfamiliar tech, upgrades, performance/security work.
- Save findings back to task/subtask.

# Documentation & Bugfix Process
- Always update related docs: READMEs, API refs, inline comments, diagrams, process docs.
- Docs must reflect behavior, interfaces, deps.
- For bugfixes: write bug report → root cause, reproduction, affected comps, fix, verification.
- Link bug report to commit/PR.

# Development Discipline
- Run tests before marking tasks done.
- Use checklists: linters, TODOs removed, security reviewed.
- Use structured logging; remove noisy logs before merging.

# Security
- Never commit secrets/keys/credentials — use `.env` + `.gitignore`.
- Sanitize user data in logs and examples.

# Collaboration
- Link relevant task/PR in code comments referencing external discussion.
- On handoff: leave written status note with next steps and blockers.

# AI / Cursor Usage
- Prefer “Suggest command” over “Run command.”
- Use research before major refactors/upgrades.
- For repetitive queries, create reusable prompts/rules.
- Do not generate dummy code/data unless asked.
- Do not open code previews; show diffs in chat only.

# Release Discipline
- Maintain `CHANGELOG.md` and update before merges/releases.

# Knowledge Management
- When solving problems or finding tools, add them to the central **Tips & Tools** doc/repo for searchability.
