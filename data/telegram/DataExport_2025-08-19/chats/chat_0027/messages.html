Hey, I just want to describe all these parameters in general:

Off-chain view:
validatorSetCaptureDelay + NETWORK_EPOCH + NETWORK_SLASHING_WINDOW (which is max(maxSlashRequestDelay + vetoDuration + maxSlashExecutionDelay)) must be less than VAULT_EPOCH_DURATION
(it means that the duration from the stake capturing timestamp until the latest possible slashing execution must be less than the vault's epoch duration)

where

validatorSetCaptureDelay, NETWORK_EPOCH, NETWORK_SLASHING_WINDOW are controlled by network

VAULT_EPOCH_DURATION, vetoDuration are controlled by vault

Your Solidity code view:
SLASHING_WINDOW (which is EPOCH_DURATION (which is validatorSetCaptureDelay + NETWORK_EPOCH) + maxSlashRequestDelay) + vetoDuration must be less than VAULT_EPOCH_DURATION

You can see that maxSlashExecutionDelay is not included in Solidity code, so this buffer must be considered either off-chain, or via some threshold to be accounted on-chain

When starting without the slashing, you may decide on numbers using e.g. such formula:
needed network epoch duration + maximum delay for slashing on your side < min(vault's epoch duration - vetoDuration)
