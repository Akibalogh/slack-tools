This is how XCLAIM works (https://eprint.iacr.org/2018/643, the base version, also implemented on Interlay - with improvements).
14:50
XCC extends this by allowing users to lock funds in a multisig with the Vault, timelocked. The Vault tracks all the deposits and must renew timelocks at the end of each round. For this, the Vault collateralizes shortly before the renewal. If this does not happen, users can force-exit out of the multisig.

This does mean, similar to DLC, that when Alice sends wrapped BTC to Bob, he only really "owns" the underlying BTC after the next commitment on Bitcoin - because we assume the worst case scenario where Alice and the Operator can collude. If Bob trusts that Operator and Alice do not collude and that the Operator does not misbehave (e.g. Operator is a network of TSS signers), then that's OK.
If Bob wants the "extra" assurance, he can request "insurance" from the Operator in the form of collateral against a fee (or Alice pays this). So, Bob's coins temporarily fall back into the fully-collateralized model (like the Interlay model right now).

We always assumed that Bob could use the non-collateralized coins in protocols that work with long-term lockups (e.g. lock up for 6 months, no collateral needed during that time since we know the BTC is locked in the multisig - just like in the DLC.Link case).
14:52
Ultimately, the difference between XCC and DLC.Link is the Vault/Operator design.

In XCC, we simply assume this as a wrapping service.
DLC.link adds the DLC part + the (ideally independent) Oracle counterparty.

Security-wise, XCC is stronger because you don't have to trust the Vault. A similar model can be added to DLC.link - but it would require collateralization of the TTP on another chain. Not sure if you need/want that though
