Regarding BitVM2 permissioned vs permissionless:

AFAIK no team in the BitVM Alliance has created a permissionless Bitcoin light client. The one that is mentioned in the paper "Bridging Bitcoin to Second Layers via BITVM2" (https://eprint.iacr.org/2025/1158.pdf) works only for static difficulty:
"we construct an on-chain Bitcoin light client for static difficulty using BITVM2-CORE". There is no possible extension of this protocol to dynamic difficulty because it uses the block height.

The old protocol "BitVM2: Bridging Bitcoin to Second Layers" (in https://bitvm.org/bitvm_bridge.pdf) uses superblocks and it's broken, but still it doesn't support varying difficulty: "our PowPV light client design currently assumes a constant Bitcoin difficulty target T".

So all existent secure BitVM-based bridges are permissioned during peg-out. Some, such as Rootstock's Union Bridge, allows anybody to join the committee, up to a maximum number of seats close to 100. So they are permissionless at certain periodic intervals, but not during peg-out.
